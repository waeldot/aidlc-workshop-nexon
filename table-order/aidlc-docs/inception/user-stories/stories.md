# 테이블오더 서비스 - User Stories

## 스토리 구성 방식
- **분류**: Feature 기반
- **세분화**: 세부 동작별 (Fine-grained)
- **수용 기준**: 혼합 (핵심 GWT + 부가 체크리스트)
- **우선순위**: 없음 (MVP 전체 필수)
- **에러 시나리오**: 별도 스토리
- **엣지 케이스**: 포함

---

## Feature 1: 태블릿 자동 로그인 및 세션 관리

### US-1.1: 태블릿 초기 설정 입력
**As a** 관리자  
**I want to** 태블릿에서 매장 식별자, 테이블 번호, 테이블 비밀번호를 입력할 수 있다  
**So that** 해당 태블릿이 특정 테이블에 귀속된다

**Acceptance Criteria:**
- Given 태블릿에 초기 설정 화면이 표시될 때
- When 관리자가 매장 식별자, 테이블 번호, 비밀번호를 입력하고 저장을 누르면
- Then 서버에 인증 요청이 전송되고 성공 시 토큰이 발급된다

추가 조건:
- 매장 식별자, 테이블 번호, 비밀번호는 필수 입력
- 입력값이 비어있으면 저장 버튼 비활성화
- 성공 시 인증 정보가 localStorage에 저장된다

---

### US-1.2: 인증 정보 로컬 저장
**As a** 시스템  
**I want to** 인증 성공 후 매장 식별자, 테이블 번호, 토큰을 localStorage에 저장한다  
**So that** 이후 앱 실행 시 자동 로그인이 가능하다

**Acceptance Criteria:**
- Given 초기 설정에서 인증이 성공했을 때
- When 토큰이 발급되면
- Then localStorage에 매장 식별자, 테이블 번호, 인증 토큰이 저장된다

추가 조건:
- 저장 데이터: storeId, tableNumber, authToken
- 브라우저 새로고침 후에도 데이터 유지

---

### US-1.3: 자동 로그인 실행
**As a** 고객  
**I want to** 태블릿 앱 실행 시 자동으로 로그인되어 메뉴 화면이 표시된다  
**So that** 별도 로그인 절차 없이 즉시 주문할 수 있다

**Acceptance Criteria:**
- Given localStorage에 유효한 인증 정보가 있을 때
- When 앱이 실행(또는 새로고침)되면
- Then 저장된 토큰으로 서버 인증을 시도하고 성공 시 메뉴 화면으로 이동한다

추가 조건:
- 로딩 중 스피너 또는 로딩 표시
- 인증 성공 시 메뉴 화면(기본 화면)으로 자동 이동

---

### US-1.4: 자동 로그인 실패 시 초기 설정 이동
**As a** 고객  
**I want to** 자동 로그인이 실패하면 초기 설정 화면으로 이동한다  
**So that** 관리자가 재설정할 수 있다

**Acceptance Criteria:**
- Given localStorage에 인증 정보가 없거나 토큰이 만료되었을 때
- When 앱이 실행되면
- Then 초기 설정 화면이 표시된다

추가 조건:
- 만료된 토큰은 localStorage에서 삭제
- 에러 메시지 표시: "인증 정보가 만료되었습니다. 관리자에게 문의하세요."

---

### US-1.5: 초기 설정 인증 실패 처리
**As a** 관리자  
**I want to** 잘못된 정보 입력 시 에러 메시지를 확인할 수 있다  
**So that** 올바른 정보로 재시도할 수 있다

**Acceptance Criteria:**
- Given 초기 설정 화면에서 정보를 입력했을 때
- When 서버 인증이 실패하면 (잘못된 매장 식별자, 테이블 번호, 비밀번호)
- Then 에러 메시지가 표시되고 입력 필드가 유지된다

추가 조건:
- 에러 메시지: "인증에 실패했습니다. 정보를 확인해 주세요."
- 입력값 유지 (재입력 편의)
- 연속 실패 시에도 재시도 가능

## Feature 2: 메뉴 조회 및 탐색

### US-2.1: 카테고리 목록 표시
**As a** 고객  
**I want to** 메뉴 카테고리 목록을 볼 수 있다  
**So that** 원하는 종류의 메뉴를 빠르게 찾을 수 있다

**Acceptance Criteria:**
- Given 메뉴 화면에 진입했을 때
- When 화면이 로드되면
- Then 모든 카테고리가 탭 또는 사이드바 형태로 표시된다

추가 조건:
- 첫 번째 카테고리가 기본 선택
- 카테고리명 표시
- 터치 친화적 크기 (최소 44x44px)

---

### US-2.2: 카테고리별 메뉴 목록 표시
**As a** 고객  
**I want to** 선택한 카테고리의 메뉴 목록을 카드 형태로 볼 수 있다  
**So that** 메뉴를 시각적으로 탐색할 수 있다

**Acceptance Criteria:**
- Given 특정 카테고리가 선택되었을 때
- When 해당 카테고리를 탭하면
- Then 해당 카테고리의 메뉴들이 카드 레이아웃으로 표시된다

추가 조건:
- 각 카드: 메뉴 이미지, 메뉴명, 가격 표시
- 카드 형태 그리드 레이아웃
- 메뉴가 없는 카테고리는 "메뉴가 없습니다" 표시

---

### US-2.3: 메뉴 상세 정보 확인
**As a** 고객  
**I want to** 메뉴 카드를 탭하면 상세 정보를 볼 수 있다  
**So that** 메뉴 설명을 확인하고 주문 여부를 결정할 수 있다

**Acceptance Criteria:**
- Given 메뉴 목록이 표시된 상태에서
- When 특정 메뉴 카드를 탭하면
- Then 메뉴 상세 정보(메뉴명, 가격, 설명, 이미지)가 표시된다

추가 조건:
- 상세 화면에서 장바구니 추가 버튼 제공
- 닫기/뒤로가기로 목록 복귀

---

### US-2.4: 카테고리 간 빠른 이동
**As a** 고객  
**I want to** 카테고리 탭을 탭하여 다른 카테고리로 즉시 이동할 수 있다  
**So that** 여러 카테고리의 메뉴를 빠르게 비교할 수 있다

**Acceptance Criteria:**
- Given 메뉴 화면에서 카테고리 탭이 표시될 때
- When 다른 카테고리 탭을 탭하면
- Then 해당 카테고리의 메뉴 목록으로 즉시 전환된다

추가 조건:
- 현재 선택된 카테고리 시각적 강조
- 전환 시 스크롤 위치 초기화

---

### US-2.5: 메뉴 로딩 실패 처리
**As a** 고객  
**I want to** 메뉴 로딩에 실패하면 에러 메시지와 재시도 버튼을 볼 수 있다  
**So that** 일시적 오류 시 다시 시도할 수 있다

**Acceptance Criteria:**
- Given 메뉴 데이터를 서버에서 가져올 때
- When 네트워크 오류 또는 서버 오류가 발생하면
- Then "메뉴를 불러올 수 없습니다" 메시지와 "다시 시도" 버튼이 표시된다

추가 조건:
- 재시도 버튼 탭 시 메뉴 재요청
- 로딩 중 스피너 표시


---

## Feature 3: 장바구니 관리

### US-3.1: 메뉴를 장바구니에 추가
**As a** 고객  
**I want to** 메뉴를 장바구니에 추가할 수 있다  
**So that** 주문할 메뉴를 모아둘 수 있다

**Acceptance Criteria:**
- Given 메뉴 상세 화면 또는 메뉴 카드에서
- When 장바구니 추가 버튼을 탭하면
- Then 해당 메뉴가 수량 1로 장바구니에 추가된다

추가 조건:
- 이미 장바구니에 있는 메뉴는 수량 +1
- 추가 성공 시 시각적 피드백 (토스트 또는 애니메이션)
- 장바구니 아이콘에 총 수량 배지 표시

---

### US-3.2: 장바구니 메뉴 수량 증가
**As a** 고객  
**I want to** 장바구니에서 메뉴 수량을 1 증가시킬 수 있다  
**So that** 같은 메뉴를 여러 개 주문할 수 있다

**Acceptance Criteria:**
- Given 장바구니에 메뉴가 있을 때
- When + 버튼을 탭하면
- Then 해당 메뉴의 수량이 1 증가하고 총 금액이 재계산된다

추가 조건:
- 수량 변경 즉시 총 금액 업데이트
- localStorage에 변경사항 즉시 반영

---

### US-3.3: 장바구니 메뉴 수량 감소
**As a** 고객  
**I want to** 장바구니에서 메뉴 수량을 1 감소시킬 수 있다  
**So that** 주문 수량을 조절할 수 있다

**Acceptance Criteria:**
- Given 장바구니에 수량 2 이상인 메뉴가 있을 때
- When - 버튼을 탭하면
- Then 해당 메뉴의 수량이 1 감소하고 총 금액이 재계산된다

추가 조건:
- 수량이 1일 때 - 버튼 탭 시 해당 메뉴 삭제
- localStorage에 변경사항 즉시 반영

---

### US-3.4: 장바구니에서 메뉴 삭제
**As a** 고객  
**I want to** 장바구니에서 특정 메뉴를 삭제할 수 있다  
**So that** 주문하지 않을 메뉴를 제거할 수 있다

**Acceptance Criteria:**
- Given 장바구니에 메뉴가 있을 때
- When 삭제 버튼을 탭하면
- Then 해당 메뉴가 장바구니에서 제거되고 총 금액이 재계산된다

추가 조건:
- 삭제 후 장바구니가 비면 "장바구니가 비어있습니다" 표시
- localStorage에 변경사항 즉시 반영

---

### US-3.5: 장바구니 전체 비우기
**As a** 고객  
**I want to** 장바구니를 한 번에 비울 수 있다  
**So that** 처음부터 다시 메뉴를 선택할 수 있다

**Acceptance Criteria:**
- Given 장바구니에 1개 이상의 메뉴가 있을 때
- When 전체 비우기 버튼을 탭하면
- Then 모든 메뉴가 장바구니에서 제거되고 총 금액이 0이 된다

추가 조건:
- 비우기 전 확인 팝업 표시 ("장바구니를 비우시겠습니까?")
- localStorage에서 장바구니 데이터 삭제

---

### US-3.6: 장바구니 총 금액 실시간 표시
**As a** 고객  
**I want to** 장바구니의 총 금액을 항상 확인할 수 있다  
**So that** 예산 내에서 주문할 수 있다

**Acceptance Criteria:**
- Given 장바구니에 메뉴가 있을 때
- When 수량이 변경되거나 메뉴가 추가/삭제되면
- Then 총 금액이 즉시 재계산되어 표시된다

추가 조건:
- 총 금액 = 각 메뉴의 (단가 × 수량) 합계
- 금액 형식: 원화 표시 (예: 15,000원)

---

### US-3.7: 장바구니 데이터 새로고침 후 유지
**As a** 고객  
**I want to** 페이지를 새로고침해도 장바구니 내용이 유지된다  
**So that** 실수로 새로고침해도 선택한 메뉴를 잃지 않는다

**Acceptance Criteria:**
- Given 장바구니에 메뉴가 담겨있을 때
- When 브라우저를 새로고침하면
- Then localStorage에서 장바구니 데이터를 복원하여 동일하게 표시된다

추가 조건:
- 메뉴명, 수량, 단가 모두 복원
- 총 금액 재계산

---

## Feature 4: 주문 생성

### US-4.1: 주문 내역 최종 확인
**As a** 고객  
**I want to** 주문 확정 전에 주문 내역을 최종 확인할 수 있다  
**So that** 실수 없이 정확한 주문을 할 수 있다

**Acceptance Criteria:**
- Given 장바구니에 메뉴가 있을 때
- When 주문하기 버튼을 탭하면
- Then 주문 확인 화면에 메뉴 목록, 수량, 단가, 총 금액이 표시된다

추가 조건:
- 각 메뉴: 메뉴명, 수량, 단가, 소계
- 하단에 총 주문 금액
- "주문 확정" 및 "돌아가기" 버튼

---

### US-4.2: 주문 확정 및 전송
**As a** 고객  
**I want to** 주문 확정 버튼을 눌러 주문을 서버에 전송할 수 있다  
**So that** 매장에서 주문을 접수받을 수 있다

**Acceptance Criteria:**
- Given 주문 확인 화면에서
- When 주문 확정 버튼을 탭하면
- Then 주문 데이터(매장ID, 테이블ID, 세션ID, 메뉴 목록, 총 금액)가 서버에 전송된다

추가 조건:
- 전송 중 로딩 표시
- 중복 전송 방지 (버튼 비활성화)

---

### US-4.3: 주문 성공 후 결과 표시
**As a** 고객  
**I want to** 주문 성공 시 주문 번호를 확인할 수 있다  
**So that** 주문이 정상 접수되었음을 알 수 있다

**Acceptance Criteria:**
- Given 주문이 서버에 성공적으로 전송되었을 때
- When 서버가 성공 응답을 반환하면
- Then 주문 번호가 표시되고 5초 후 메뉴 화면으로 자동 리다이렉트된다

추가 조건:
- 주문 번호 크게 표시
- "주문이 접수되었습니다" 메시지
- 5초 카운트다운 표시
- 장바구니 자동 비우기 (localStorage 포함)

---

### US-4.4: 주문 실패 시 에러 처리
**As a** 고객  
**I want to** 주문 실패 시 에러 메시지를 확인하고 재시도할 수 있다  
**So that** 주문을 다시 시도할 수 있다

**Acceptance Criteria:**
- Given 주문 확정 버튼을 눌렀을 때
- When 서버 오류 또는 네트워크 오류가 발생하면
- Then 에러 메시지가 표시되고 장바구니 데이터는 유지된다

추가 조건:
- 에러 메시지: "주문에 실패했습니다. 다시 시도해 주세요."
- 장바구니 데이터 보존 (삭제하지 않음)
- 재시도 버튼 또는 주문 확정 버튼 재활성화

---

### US-4.5: 빈 장바구니 주문 방지
**As a** 고객  
**I want to** 장바구니가 비어있으면 주문할 수 없다  
**So that** 빈 주문이 전송되지 않는다

**Acceptance Criteria:**
- Given 장바구니가 비어있을 때
- When 주문하기 버튼 영역을 확인하면
- Then 주문하기 버튼이 비활성화되어 있다

추가 조건:
- 비활성화 상태 시각적 표시 (회색 처리)
- "메뉴를 추가해 주세요" 안내 문구


---

## Feature 5: 주문 내역 조회 (고객)

### US-5.1: 현재 세션 주문 목록 조회
**As a** 고객  
**I want to** 현재 테이블 세션의 주문 내역을 볼 수 있다  
**So that** 이번 방문에서 주문한 내용을 확인할 수 있다

**Acceptance Criteria:**
- Given 주문 내역 화면에 진입했을 때
- When 화면이 로드되면
- Then 현재 세션의 주문만 시간 순으로 표시된다

추가 조건:
- 이전 세션(이용 완료된) 주문은 표시하지 않음
- 주문별: 주문 번호, 주문 시각, 메뉴 및 수량, 금액, 상태
- 주문이 없으면 "주문 내역이 없습니다" 표시

---

### US-5.2: 주문 상태 실시간 업데이트
**As a** 고객  
**I want to** 주문 상태가 변경되면 실시간으로 반영된다  
**So that** 음식 준비 상황을 알 수 있다

**Acceptance Criteria:**
- Given 주문 내역 화면을 보고 있을 때
- When 관리자가 주문 상태를 변경하면 (대기중→준비중→완료)
- Then SSE를 통해 해당 주문의 상태가 실시간으로 업데이트된다

추가 조건:
- 상태 변경 시 시각적 피드백 (색상 변경 또는 애니메이션)
- 상태: 대기중(노란색), 준비중(파란색), 완료(초록색)

---

### US-5.3: 주문 내역 로딩 실패 처리
**As a** 고객  
**I want to** 주문 내역 로딩 실패 시 에러 메시지와 재시도 버튼을 볼 수 있다  
**So that** 일시적 오류 시 다시 시도할 수 있다

**Acceptance Criteria:**
- Given 주문 내역을 서버에서 가져올 때
- When 네트워크 오류가 발생하면
- Then "주문 내역을 불러올 수 없습니다" 메시지와 재시도 버튼이 표시된다

---

### US-5.4: SSE 연결 끊김 시 자동 재연결
**As a** 고객  
**I want to** SSE 연결이 끊어지면 자동으로 재연결된다  
**So that** 수동 새로고침 없이 실시간 업데이트를 계속 받을 수 있다

**Acceptance Criteria:**
- Given SSE 연결이 활성화된 상태에서
- When 네트워크 일시 단절로 연결이 끊기면
- Then 자동으로 재연결을 시도하고 성공 시 업데이트를 재개한다

추가 조건:
- 재연결 시도 간격: 점진적 증가 (1초, 2초, 4초...)
- 최대 재시도 횟수 후 "연결이 끊어졌습니다. 새로고침해 주세요." 표시

추가 조건:
- 토큰 유효성 검증 (만료 시간 확인)

### US-6.3: 관리자 자동 로그아웃

추가 조건:
- localStorage에서 만료된 토큰 삭제

추가 조건:

**Acceptance Criteria:**
- Given 관리자가 이용 완료를 요청한 시점에
- When 동시에 고객이 새 주문을 전송하면
- Then 이용 완료가 우선 처리되고, 새 주문은 실패 응답을 받는다

추가 조건:
- 고객에게 "세션이 종료되었습니다" 에러 메시지 표시
- 고객 화면은 초기 상태로 리셋
- 데이터 정합성 보장 (트랜잭션 처리)

---

### US-9.2: 동시 주문 처리
**As a** 시스템  
**I want to** 같은 테이블에서 동시에 여러 주문이 들어와도 모두 정상 처리된다  
**So that** 주문이 누락되지 않는다

**Acceptance Criteria:**
- Given 같은 테이블에서
- When 짧은 시간 내에 여러 주문이 동시에 전송되면
- Then 모든 주문이 순차적으로 처리되어 저장된다

추가 조건:
- 각 주문에 고유 주문 번호 부여
- 총 주문액 정확히 합산
- 관리자 대시보드에 모든 주문 표시

---

### US-9.3: 세션 종료 후 고객 화면 초기화
**As a** 고객  
**I want to** 관리자가 이용 완료를 처리하면 내 화면이 초기화된다  
**So that** 다음 고객이 이전 주문을 볼 수 없다

**Acceptance Criteria:**
- Given 고객이 주문 내역 또는 메뉴 화면을 보고 있을 때
- When 관리자가 해당 테이블의 이용 완료를 처리하면
- Then SSE를 통해 세션 종료 이벤트를 수신하고 주문 내역이 비워진다

추가 조건:
- 장바구니도 비워짐
- 메뉴 화면으로 자동 이동
- "새로운 이용이 시작되었습니다" 안내 (선택)

## Feature 10: 메뉴 Seed 데이터

### US-10.1: 초기 메뉴 데이터 로드
**As a** 시스템  
**I want to** 애플리케이션 시작 시 Seed 데이터로 메뉴가 초기화된다  
**So that** 별도 메뉴 등록 없이 서비스를 시작할 수 있다

**Acceptance Criteria:**
- Given 데이터베이스가 비어있을 때
- When 애플리케이션이 시작되면
- Then Seed 데이터에 정의된 카테고리와 메뉴가 DB에 삽입된다

추가 조건:
- 카테고리: 최소 3개 이상
- 메뉴: 카테고리당 최소 3개 이상
- 각 메뉴: 메뉴명, 가격, 설명, 카테고리, 이미지 URL
- 이미 데이터가 있으면 Seed 실행하지 않음 (멱등성)
